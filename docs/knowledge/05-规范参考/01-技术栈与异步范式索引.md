# 技术栈与异步范式索引

## 1. 当前技术栈

| 类别 | 当前选型 | 说明 |
| --- | --- | --- |
| 语言 | Java 17 | 由 `pom.xml` 中的 `java.version` 指定 |
| 构建工具 | Maven | 负责依赖管理、测试与打包 |
| 后端框架 | Spring Boot 3.5.13 | 应用启动和自动配置核心 |
| Web 框架 | Spring MVC | 提供 REST API |
| 样板代码工具 | Lombok | 用于生成 getter、setter、构造器等 |
| 测试框架 | Spring Boot Test、JUnit 5 | 当前主要用于上下文启动测试 |
| 前端实现 | 原生 HTML/CSS/JavaScript | 静态页面直接托管在应用中 |
| 数据存储 | 内存 `ConcurrentHashMap` | 当前无数据库 |

## 2. 当前处理范式

当前项目使用的是经典同步请求处理模型：

1. 浏览器发起 HTTP 请求
2. 控制器同步执行
3. 服务层同步处理
4. 结果立即返回前端

这个项目当前没有：

- `@Async`
- WebFlux
- 消息队列
- 定时任务编排
- 后台任务系统

## 3. 当前并发相关实现

虽然项目整体是同步 API，但 `SkillService` 里已经使用了并发安全的数据结构：

- `ConcurrentHashMap`
- `AtomicLong`

这说明在多个请求并发访问时，内存存储和 ID 分配不会立刻出现最基础的线程安全问题。

但要注意，这不等于项目已经完成了复杂并发设计。当前只解决了“简单的内存并发访问安全”，没有解决持久化一致性、事务、分布式场景等问题。

## 4. 当前不需要异步的原因

目前每个请求做的事情都很轻：

- 查询内存数据
- 更新内存数据
- 返回 JSON

在这种场景下，引入异步体系不会带来明显收益，反而会增加复杂度。

## 5. 如果后续要引入异步，推荐场景

以下场景出现时，再考虑异步会更合适：

- Skill 内容需要做耗时解析
- 需要导入或导出大量文档
- 需要调用外部大模型或第三方接口
- 需要记录审计日志且不想阻塞主请求

## 6. 如果后续要引入异步，推荐顺序

1. 优先考虑最轻量的 `@Async`
2. 再考虑任务队列或消息队列
3. 只有在响应式需求明确时，再考虑 WebFlux

## 7. 当前项目的技术边界总结

- 这是一个轻量、同步、单体、内存存储的 Spring Boot 项目
- 当前重点在于功能通路打通，而不是复杂基础设施
- 后续扩展时，应基于真实需求逐步引入数据库、异步和模块拆分，而不是一次性堆叠技术复杂度
