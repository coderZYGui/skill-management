# 模块依赖与分层建议

## 1. 当前依赖关系

当前项目的依赖关系非常直接，核心链路可以概括为：

- `index.html` 依赖后端 HTTP API `/api/skills`
- `SkillController` 依赖 `SkillService`
- `SkillController` 同时依赖 `Skill` 和 `Result`
- `SkillService` 依赖 `Skill`
- `SkillService` 依赖 JDK 并发和时间类，如 `ConcurrentHashMap`、`AtomicLong`、`LocalDateTime`
- `TestDemoApplication` 依赖 Spring Boot 自动配置体系

## 2. 当前依赖方向是合理的部分

- 控制器不直接管理存储结构，而是通过服务层处理业务。
- `Result<T>` 作为通用返回对象，被控制层统一复用。
- 页面通过 HTTP API 访问后端，没有直接耦合 Java 类。
- `Skill` 作为领域对象，在控制层和服务层之间传递，结构简单直观。

## 3. 当前分层上还比较薄弱的部分

- 没有 Repository 或 DAO 层，存储细节直接写在 `SkillService` 中。
- 控制器内存在较多 `try/catch` 和日志处理逻辑，后续可以下沉到统一异常处理层。
- `Skill` 同时承担请求体、响应体和领域模型角色，后续复杂化后可能需要拆 DTO。
- 时间字段当前用 `String` 保存，便于展示但不利于更复杂的时间计算。

## 4. 当前建议的层次边界

为了让后续功能扩展时不失控，建议继续遵守以下边界：

- Controller 只负责协议转换、请求接收、调用服务、返回响应。
- Service 负责业务规则和数据编排。
- Model 负责表达领域对象，不承担控制流程。
- Common 目录放通用响应对象、异常对象、工具类等。
- Static 页面只通过 API 与后端交互，不要嵌入后端实现细节。

## 5. 当前项目的推荐依赖规则

- `controller` 可以依赖 `service`、`model`、`common`
- `service` 可以依赖 `model`，必要时依赖 `common`
- `model` 不应依赖 `controller` 或 `service`
- `common` 尽量不反向依赖业务模块
- 前端页面只能通过 HTTP 访问业务能力

## 6. 如果项目继续扩展，优先建议补齐的层

### 6.1 Repository 层

如果后续要接数据库，优先把内存存储逻辑从 `SkillService` 中抽离到 Repository 层，避免服务层同时承担业务和存储职责。

### 6.2 DTO 与参数校验

如果前端字段继续增多，建议引入：

- `SkillCreateRequest`
- `SkillUpdateRequest`
- `SkillResponse`
- `@Valid` + `jakarta.validation`

### 6.3 全局异常处理

可以引入 `@RestControllerAdvice` 统一处理参数错误、系统异常和业务异常，避免每个控制器重复 `try/catch`。

### 6.4 持久化层

如果目标不再是演示项目，建议尽快接入数据库，否则所有 Skill 数据都会在应用重启后丢失。

## 7. 当前阶段的结论

以“原型应用”的标准看，当前依赖关系是清晰的；以“可持续扩展”的标准看，项目还处于非常早期，后续最值得优先补强的是持久化、参数校验、统一异常处理和层次拆分。
