# 跨模块通信与 ModuleRegistry 说明

## 1. 为什么会有这篇文档

当前仓库文件名里包含 `ModuleRegistry`，但要先明确一个事实：项目现在还没有真正实现一个独立的 `ModuleRegistry` 类，也没有拆成多个业务模块。

因此，这篇文档的重点不是“当前已有的注册中心实现”，而是说明：

- 当前项目实际上如何通信
- 为什么目前不需要 `ModuleRegistry`
- 如果后续拆模块，推荐怎样设计

## 2. 当前项目的真实通信方式

### 2.1 页面到后端

- 通信方式：HTTP
- 发起方：`static/index.html`
- 接收方：`SkillController`
- 接口前缀：`/api/skills`

这是当前最明显的跨边界通信方式，也是用户真正可见的主业务链路。

### 2.2 控制器到服务

- 通信方式：Spring 依赖注入 + 直接方法调用
- 发起方：`SkillController`
- 接收方：`SkillService`

这是单体应用内部最常见的调用模式，当前已经足够简单直接。

### 2.3 服务到存储

- 通信方式：直接访问内存 Map
- 发起方：`SkillService`
- 接收方：`skillStore`

这里还没有抽象出存储接口，因此也谈不上模块注册。

## 3. 为什么当前不需要 ModuleRegistry

当前项目只有一个核心业务域对象 `Skill`，并且业务规模很小：

- 没有多个业务子系统
- 没有插件式模块装配
- 没有事件总线
- 没有服务发现
- 没有跨模块接口治理需求

在这种情况下，如果强行引入 `ModuleRegistry`，会增加抽象成本，却没有明显收益。

## 4. 如果后续扩展，ModuleRegistry 可以解决什么问题

当项目继续演进并出现以下情况时，可以考虑引入模块注册机制：

- 出现多个领域模块，例如 `skill`、`template`、`user`、`audit`
- 需要按模块名动态查找处理器
- 需要让一个模块能力被其他模块统一发现
- 需要做插件式扩展，而不是在控制器中硬编码分支

## 5. 推荐的演进思路

### 5.1 先接口化，再注册化

建议先定义模块能力接口，例如：

```java
public interface SkillModule {
    String moduleName();
}
```

然后再通过 Spring 注入一组实现类，最后由统一的注册中心管理。

### 5.2 推荐的注册中心职责

如果未来需要 `ModuleRegistry`，它的职责建议只有这些：

- 在启动时收集所有模块实现
- 提供按模块名查询能力
- 统一做重复注册校验
- 为其他模块提供只读访问入口

不建议它承担业务编排、事务控制或页面路由职责。

## 6. 结合当前项目的落地建议

如果未来继续扩展 `Skill` 管理系统，可以按下面顺序演进：

1. 先拆出 Repository 层。
2. 再把预置模板能力从页面脚本中抽离成独立后端模块。
3. 如果出现多个能力提供者，再考虑 `ModuleRegistry`。

## 7. 当前结论

当前项目已经存在“通信”，但还没有“模块注册”。在现阶段，最务实的做法是继续使用 Spring 依赖注入和 REST API；只有当项目真的演进为多能力模块或插件式结构时，再引入 `ModuleRegistry` 才更合适。
