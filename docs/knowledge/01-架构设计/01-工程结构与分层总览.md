# 工程结构与分层总览

## 1. 当前工程形态

`skill-management` 当前是一个单模块 Maven 工程，整体以 Spring Boot 为启动容器，后端通过 Spring MVC 提供 REST API，前端使用 `static/index.html` 直接由应用托管。

这个项目更接近“轻量的 Skill 管理原型系统”，而不是完整的企业级业务系统。它已经具备一套完整的前后端联动链路，但还没有接入持久化、鉴权和复杂模块拆分。

## 2. 当前目录结构

```text
skill-management/
├─ AGENTS.md
├─ docs/
│  └─ knowledge/
├─ pom.xml
├─ src/
│  ├─ main/
│  │  ├─ java/com/xun/
│  │  │  ├─ TestDemoApplication.java
│  │  │  ├─ common/Result.java
│  │  │  ├─ controller/SkillController.java
│  │  │  ├─ model/Skill.java
│  │  │  └─ service/SkillService.java
│  │  └─ resources/
│  │     ├─ application.properties
│  │     └─ static/index.html
│  └─ test/
│     └─ java/com/xun/TestDemoApplicationTests.java
└─ target/
```

说明：

- `AGENTS.md` 是仓库总导航。
- `docs/knowledge` 是项目知识库。
- `src/main/java` 是后端代码主目录。
- `src/main/resources/static` 是静态前端页面目录。
- `target` 是编译产物目录，不应作为业务代码理解入口。

## 3. 当前分层结构

### 启动层

- 文件：`TestDemoApplication.java`
- 作用：应用启动入口，负责加载 Spring Boot 自动配置并启动内嵌 Web 容器。

### 控制层

- 文件：`controller/SkillController.java`
- 作用：对外暴露 `/api/skills` REST 接口。
- 职责：
  - 接收 HTTP 请求
  - 记录日志
  - 调用 `SkillService`
  - 将业务结果统一包装成 `Result<T>`

### 服务层

- 文件：`service/SkillService.java`
- 作用：封装 Skill 的增删改查逻辑。
- 职责：
  - 维护内存存储 `ConcurrentHashMap<Long, Skill>`
  - 分配自增 ID
  - 写入创建时间和更新时间
  - 返回当前 Skill 记录

### 领域模型层

- 文件：`model/Skill.java`
- 作用：定义 Skill 的数据结构。
- 核心字段：
  - `id`
  - `name`
  - `displayName`
  - `description`
  - `content`
  - `author`
  - `category`
  - `createdAt`
  - `updatedAt`

### 通用对象层

- 文件：`common/Result.java`
- 作用：定义统一响应结构。
- 响应格式：
  - `code`
  - `message`
  - `data`

### 前端展示层

- 文件：`resources/static/index.html`
- 作用：提供 Skill 管理页面。
- 已实现的页面能力：
  - 展示 Skill 列表
  - 新增 Skill
  - 编辑 Skill
  - 删除 Skill
  - 查看详情弹窗
  - 通过预置模板快速填充表单

### 测试层

- 文件：`TestDemoApplicationTests.java`
- 作用：校验 Spring 容器是否可正常启动。
- 当前测试范围很小，只有基础的上下文加载测试。

## 4. 请求链路

当前项目最重要的主链路如下：

1. 用户打开 `/index.html` 或根路径静态页面。
2. 页面中的 JavaScript 使用 `fetch` 调用 `/api/skills`。
3. `SkillController` 接收请求并调用 `SkillService`。
4. `SkillService` 从内存 `skillStore` 中读取或写入 `Skill` 数据。
5. 控制器将结果包装为 `Result<T>` 返回给前端。
6. 前端收到 JSON 后刷新表格、弹窗或提示信息。

## 5. 当前系统的关键特征

- 单体应用：前后端共部署、共启动。
- 内存存储：所有数据保存在进程内存里，重启即清空。
- 同步处理：请求处理链路是同步的，没有异步任务编排。
- 轻量原型：更适合演示、练习和后续扩展，不适合直接用于生产场景。

## 6. 当前已知限制

- 没有数据库，因此不存在持久化能力。
- 没有参数校验注解，非法输入主要依赖前端校验和运行时行为。
- 没有统一异常处理器，异常由控制器本地 `try/catch` 处理。
- 没有 Repository 层，服务直接管理内存 Map。
- 页面中文文字和部分日志出现乱码，说明源码编码还有待统一。
