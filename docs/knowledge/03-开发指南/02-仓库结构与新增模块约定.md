# 仓库结构与新增模块约定

## 1. 当前仓库结构

当前仓库主要包含以下几类内容：

- `src/main/java/com/xun`：后端业务代码
- `src/main/resources`：配置与静态资源
- `src/test/java/com/xun`：测试代码
- `docs/knowledge`：项目知识库
- `AGENTS.md`：知识总入口

## 2. 当前代码落位约定

- 启动类放在根包 `com.xun`
- 控制器放在 `com.xun.controller`
- 服务类放在 `com.xun.service`
- 领域模型放在 `com.xun.model`
- 通用对象放在 `com.xun.common`
- 页面文件放在 `src/main/resources/static`

## 3. 新增功能时的推荐步骤

以“新增一个新的业务对象”为例，建议按以下顺序扩展：

1. 先定义领域模型
2. 再补服务类
3. 再补控制器
4. 再补页面交互
5. 最后补测试和知识库文档

## 4. 当前项目推荐遵守的实现约定

- REST 路径使用复数资源名，例如 `/api/skills`
- 控制器返回值统一使用 `Result<T>`
- 服务层负责业务行为，不要把页面逻辑写进服务层
- 页面只通过 HTTP 调用后端，不直接依赖 Java 代码
- 文档新增时同步更新所属目录 `README.md` 和顶层 `AGENTS.md`

## 5. 如果后续新增 Repository 层

虽然当前还没有 Repository 层，但如果后续接数据库，建议新增如下目录：

```text
src/main/java/com/xun/repository
```

并把 `SkillService` 中的内存存储逻辑逐步迁移过去。

## 6. 如果后续新增前端资源

当前页面是单文件 `index.html`。如果前端交互继续复杂化，建议进一步拆分：

- `static/js`
- `static/css`
- `static/assets`

这样可以避免所有样式和脚本堆叠在一个 HTML 文件中。

## 7. 文档维护约定

每次新增或明显修改功能时，建议同步更新以下文档：

- 影响整体理解时，更新 `AGENTS.md`
- 影响某个知识分区时，更新对应目录的 `README.md`
- 新增架构、模块、开发或排查知识时，补对应专题文档

## 8. 当前不建议做的事

- 不建议在控制器里直接操作存储 Map
- 不建议跳过服务层直接让页面设计决定后端模型
- 不建议把编译产物 `target` 目录提交到版本库
- 不建议在文档中假定项目已经接入数据库或多模块能力
